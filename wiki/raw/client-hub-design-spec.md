## Problem Statement

Monday.com is the backbone for PPM's project management and client communication, but critical features are gated behind enterprise pricing:
- **Client-facing portals** — enterprise only
- **Comment/feedback routing** — can't transfer from client boards to internal tools automatically
- **Mirrored column visibility** — restricted to board members, can't share cross-board data
- **Board sprawl** — too many boards for what should be a simple 3-layer hierarchy

The result: "Honestly it's chaos — things fall through the cracks regularly." Specifically:
1. Missed deliverable deadlines because internal status didn't reach the right person
2. Client feedback that never made it back to the creative team

Monday serves 4 functions: task tracking, client communication, planning, and scope tracking against client retainers. Employees and clients all touch it daily. Current cost: $450/month, and the features PPM needs would require an enterprise upgrade.

### Mirror Column Architecture (the fragile dependency)
The entire cross-board visibility system relies on manual board relations + mirror columns:
- Per-client Creative Request boards mirror Design Status and Link to Review Asset FROM Creative Pipeline
- Per-client Website/Ops boards mirror Internal Status, Figma URL, Preview URL FROM central Website/Ops board
- Client Master Board mirrors Estimated Hours and Actual Hours FROM pipeline boards
- **Failure mode:** If someone forgets to manually link a board relation, mirrors show blank → account managers lose visibility

**In Worklenz fork:** RLS + single-project-per-client eliminates this entirely. Tasks belong to a client by default — no manual linking, no mirrors, no silent failures.

### Pain Points Confirmed
1. **Manual board relation linking** — every Creative Pipeline item must be manually linked to its per-client Creative Request. Forgotten link = invisible status
2. **Creative brief re-entry** — briefs entered by account managers in per-client boards are copy-pasted into Creative Pipeline items
3. **No cross-board view** — account managers cross-reference 2-3 boards to see one client's full picture
4. **Hour tracking gaps** — new items not linked to Client Master Board don't roll up to utilization formulas
5. **Two parallel systems** — old per-client boards and new consolidated boards with no migration path

**Primary users:**
- PPM internal team (account managers, creative team, ops) — need internal execution boards
- PPM clients — need a portal to see deliverable status, provide feedback, approve work
- Jon (agency owner) — needs master view across all clients + scope tracking

## Constraints

1. All 4 functions (task tracking, client comms, planning, scope tracking) in one workflow source of truth — planning and scope tracking plug in as features, not separate tools
2. Clients need their own branded portal — can't share internal boards
3. Must integrate with existing ecosystem (CreativeHQ, Data Warehouse, Creative Pipeline)
4. Cost-effective — no enterprise paywall, self-hosted preferred
5. Must be usable by non-technical team members and clients on day 1 — can't feel like a prototype
6. AGPLv3 license compliance — fork repo must remain public on GitHub to satisfy source disclosure for hosted client portal

## Premises

1. **One workflow source of truth with the 3-layer model** — planning and scope tracking are features within it, not separate tools (revised from Codex challenge)
2. Clients need their own portal/view — sharing internal boards is not acceptable
3. Must integrate natively with existing ecosystem tools
4. Cost matters — self-hosted open-source fork eliminates recurring SaaS costs ($450/month + enterprise upgrade)
5. The core problem is handoff routing, not PM features — Worklenz provides the PM table stakes, the 3-layer model provides the handoff guarantees

## Fork Strategy — Soft Fork with Upstream Tracking

### Code Separation Rules

ALL PPM customizations follow these conventions:

**Database:**
- New tables → prefix: `ppm_` (e.g., ppm_deliverables, ppm_retainers, ppm_client_users)
- New columns → NEVER modify existing Worklenz tables. Create ppm_ extension tables with FK to Worklenz tables
- New migrations → directory: `migrations/ppm/` (separate from Worklenz migrations/)
- New functions → prefix: `ppm_` (e.g., ppm_route_status_change)

**Backend:**
- New routes → directory: `src/ppm/`
- New middleware → directory: `src/ppm/middleware/`
- Modified routes → AVOID. If unavoidable, wrap with PPM-OVERRIDE pattern:
  `// PPM-OVERRIDE: [ticket] [reason]`
  `// ORIGINAL: [original code commented, not deleted]`

**Frontend:**
- New components → directory: `src/app/ppm/`
- Theme overrides → `src/app/ppm/theme/` (CSS variables, Ant Design config)
- Modified components → same PPM-OVERRIDE pattern

**Config:**
- PPM env vars → prefix: `PPM_` (e.g., PPM_CLIENT_PORTAL_ENABLED)

### Upstream Sync Cadence

**Weekly (automated):**
- `git fetch upstream`
- Compare upstream/main with last synced commit
- Generate diff summary → post to Slack via OpenClaw

**Monthly (human review, ~30 min):**
- Review accumulated upstream changes
- Categorize: SECURITY | BUG_FIX | FEATURE | BREAKING
- Cherry-pick SECURITY immediately
- Cherry-pick BUG_FIX if relevant
- SKIP FEATURE unless compelling
- Flag BREAKING for investigation

**Quarterly (architecture review, ~1 hour):**
- Full upstream diff review
- Assess divergence: LOW (<20 files) | MEDIUM | HIGH
- If HIGH: trigger kill criteria reassessment
- Update UPSTREAM_STATUS.md

### Cherry-Pick Workflow

```
git checkout -b upstream-sync/YYYY-MM-DD
git cherry-pick <commit-hash>
# Run full test suite (Worklenz + PPM) against staging DB
# PR with label "upstream-sync", review, merge
# Log in UPSTREAM_LOG.md
```

### Tracking Files (in repo root)

- `UPSTREAM_STATUS.md` — last synced commit, divergence level, known conflicts
- `UPSTREAM_LOG.md` — every cherry-pick: date, commit, category, conflicts
- `PPM_EXTENSIONS.md` — every file PPM has added or modified (auto-generated)

## Phase 1: Kill-Shot Spike (~3-5 days CC time)

Prove the hard parts work before committing to full Phase 2. Each step validates a core assumption.

- **1a.** Fork Worklenz repo, deploy vanilla to Railway — prove it runs
- **1b.** Add one `ppm_` table, enable RLS, verify client isolation works at DB level
- **1c.** Build one PPM route with LISTEN/NOTIFY routing — prove the routing pattern works without PG triggers on Worklenz tables
- **1d.** Build one client portal page with magic link auth — prove client-facing views work with Passport.js sessions
- **1e.** Quick API write test — create a task via Worklenz's API programmatically (validates PPMBot integration path)
- **1f.** Verify Worklenz time tracking supports increment-based input (15-min intervals), not just timer-based — if not, scope the PPM extension needed
- **1g.** **GO/NO-GO decision:** If 1b-1f work cleanly without touching >5 Worklenz core files → proceed to Phase 2. If any require heavy core modification → trigger kill criteria.

### Module Disposition (from eng review)

Disable/hide unnecessary modules — do NOT delete them (easier upstream merges):
- Keep: tasks, boards, assignments, time tracking, notifications, files
- Disable: any modules with no mapping to 3-layer model or ecosystem integrations

## Phase 2: 3-Layer Customization (~1-2 weeks CC time)

### Architecture (revised from eng review + CEO review + TaskFlow)

- **Single project per client** with role-based visibility filters (not 3 linked projects)
- **Task types** within each project: `creative`, `web_ops`, `retention`, `general` (maps to Monday's per-client Creative Requests + Website/Ops + CRM/Retention boards)
- **Admin-managed dropdown options** for priority, channel, and type — stored in DB, not hardcoded enums (from TaskFlow). Admins can add/remove values without code changes.
- **Session auth** via Passport.js (Worklenz's existing auth — no JWT swap needed)
- **Hybrid stored functions:** New PPM features in TypeScript, touch existing PL/pgSQL only when required
- **Routing via LISTEN/NOTIFY** (not PG triggers → queue → cron)
- **Client isolation via PostgreSQL RLS** (not per-route filtering alone)
- **Cross-client internal views** — internal team sees aggregated kanban across all clients' tasks (replaces Monday's Creative Pipeline board)
- **Calendar view** — monthly grid plotting tasks by send date, filterable by client (from TaskFlow)

**Internal-only states:** `queued`, `in_progress`, `internal_review`, `revision`. Clients see: `client_review`, `approved`, `done`, plus a generic "in progress" for all internal states. Retainer time data follows the same visibility rules — internal layer shows raw hours, client layer shows approved summaries only (visibility controlled by 3-layer model).

**Revision loop (from TaskFlow):** When a client rejects work (`client_review` → `revision`), the task routes back to the assigned employee with the client's structured feedback attached. The employee reworks and resubmits, cycling back to `client_review`. This loop can repeat until the client approves. Each cycle is logged in `ppm_audit_log` with the feedback details.

**Admin ↔ Client feedback loop (from TaskFlow v2):** When a task is `queued`, the admin can approve & assign OR send back to the client with structured feedback. The feedback modal requires selecting a predefined reason:

| Reason | Description |
|--------|-------------|
| Missing items | Required form fields not filled out |
| Unclear scope | Description needs more detail for the team to execute |
| Wrong channel/type | Client selected incorrect dropdown values |
| Duplicate request | Task already submitted |
| Out of scope | Not covered by current retainer agreement |
| Budget exceeded | Retainer hours for the current period are depleted |
| Needs assets | Reference files, brand assets, or links missing |
| Other | See comment for details |

These reasons should be seeded as initial data but are admin-manageable (add/remove via UI, similar to dropdown options).

**3-tier visibility model (from TaskFlow v2):** Statuses have three visibility tiers, not two. Internal (admin + employee only), Shared (all parties see same label), and Client (client-facing). Client-facing dashboard labels are sanitized:

| Status | Visibility | Client Sees |
|--------|-----------|-------------|
| `queued` | Internal | "Submitted" |
| `in_progress` | Internal | "In Progress" |
| `internal_review` | Internal | "In Progress" |
| `client_review` | Client | "Awaiting Review" |
| `revision` | Shared | "Revision" |
| `approved` | Client | "Approved" |
| `done` | Client | "Done" |

Dashboard stat cards also use per-role labels: admin sees "Queued" / "Active" / "Client Review" / "Approved"; clients see "Submitted" / "Active" / "Awaiting Review" / "Approved".

### Routing Mechanism — LISTEN/NOTIFY

```
PPM Route Handler (status change)
        │
        ├── Update ppm_deliverables.status
        ├── INSERT into ppm_routing_log (sequence_number, pending)
        └── NOTIFY ppm_routing, '{"deliverable_id": "...", "new_status": "...", "seq": N}'
                │
                ▼
PPM Routing Service (LISTEN ppm_routing)
        │
        ├── Process in sequence_number order per deliverable
        ├── Apply routing rules (see table below)
        ├── Update ppm_routing_log.status → completed/failed
        └── Log to ppm_audit_log
```

| Event | Action | Notify |
|-------|--------|--------|
| Task submitted by client | Create task in pending state | Admin notified |
| Admin approves task | Status → queued, ready for assignment | — |
| Admin assigns to employee | Status → assigned | Employee notified |
| Employee starts work | Status → in_progress | — |
| Employee submits for review | Status → client_review, visibility → client_visible | Client + admin notified |
| Client → `approved` | Status → done, set month_completed, close action items | Employee + admin notified |
| Client → `revision` (reject) | Status → revision, attach feedback, route back to employee | Employee + admin notified |
| Comment posted on task | — | All other parties on the task notified |
| Any status change | Log to audit, update master rollup | — |

### Comment System UX (from TaskFlow)

- Slide-out panel per task with chat-style messages
- Role color-coding: **purple** (admin), **blue** (client), **green** (employee)
- Sender avatar (initials), name, role badge, timestamp
- Own messages align right, others align left
- All parties on a task (client, admin, assigned employee) can read and post
- New comment triggers notification to all other parties

### Client Portal Auth

- **Provisioning:** Internal admin invites client user by email → magic link sent
- **Auth flow:** Magic link → session created via Passport.js. Optional password setup for returning users.
- **Scoping:** Middleware sets `SET LOCAL ppm.current_client_id = '<id>'` on each request → RLS enforces isolation at DB level
- **UI branching:** Client users get role-specific views:
  - **Landing:** Action Queue — "here's what needs your attention" (inbox model)
  - **Creative preview:** Hybrid — thumbnail inline + "View Full Size" link to source
  - **Feedback:** Structured revision form — checkboxes for change type + text field + urgency selector
  - **Calendar:** Monthly view of their tasks by send date
  - **Branding:** Per-client logo and colors loaded from `ppm_clients.branding_config`
- **Role-specific home screens:**
  - Client users → Action Queue
  - Team members → My Tasks (Worklenz default)
  - Jon → Master Overview (cross-client dashboard)
- **Offboarding:** Admin deactivates client user → `deactivated_at` set → session rejected

### Scope Tracking

Retainer rollup on Master board: `SUM(actual_hours) / budgeted_hours` per client per period. Color-coded: green (<80%), yellow (80-100%), red (>100%). Retainer values entered by admin in client settings. Internal layer shows raw hours + billable flags; client layer shows approved summaries only (visibility controlled by 3-layer model). `month_completed` field enables filtering hours by completion month for retainer period reporting.

### Error Rescue Specifications

| Error | Rescue | User Sees |
|-------|--------|-----------|
| FK violation during routing (deliverable deleted mid-route) | Mark routing_log entry FAILED, log context, continue batch. Alert if >3 FK failures in 10min. | Nothing (async) |
| Connection pool exhaustion (worker + app compete) | Separate pools: main=15, worker=5. Retry 3x with 1s/2s/4s backoff. Pause worker 30s if exhausted. Health check at /ppm/health. | Nothing (transparent) |
| Routing queue backlog | Monitor depth every cycle. Warning >50 for 2min. Alert >200 for 5min via Slack webhook. | Client sees "Status updating..." instead of stale silence |
| Orphan time entries (no client_id) | NOT NULL constraint on client_id. Unallocated entries shown as "X hours unallocated" in yellow on retainer dashboard. Weekly admin notification. | Visible yellow indicator, not silent omission |
| Magic link expired | "Link expired" message + one-click resend | Clear message + action |
| Client accesses wrong client's data | RLS blocks at DB level. Log + deny + alert admin. | 403 Forbidden |
| Rapid status changes (2 changes in <10s) | Sequence numbers on routing_log. Process in order. Last-write-wins if later sequence already processed. | Correct final state |

## AGPL-3.0 Compliance

Confirmed 2026-03-25. The fork repo must remain public on GitHub to satisfy AGPL-3.0 Section 13 source disclosure obligations for the hosted client portal.

**Requirements:**
1. Public GitHub repo, kept in sync with deployed version (CI/CD must push before deploy)
2. "Source Code" link in portal footer pointing to the fork
3. All `ppm_` code is AGPL-3.0 too — same process = same work (directory separation is not a legal boundary)
4. Preserve Worklenz copyright notices and "Powered by Worklenz" attribution

**Escape hatch:** Proprietary code can live in a separate service communicating via REST API. Different process, different repo = not a derivative work.
