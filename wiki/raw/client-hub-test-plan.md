## Affected Pages/Routes

### Client Portal (NEW)
- `/portal/login` — client_user magic link or password login
- `/portal/deliverables` — filtered task list (client-visible states only)
- `/portal/deliverables/:id` — single deliverable with approve/reject/comment
- `/portal/files` — file attachments scoped to client

### Internal Extensions (MODIFIED)
- `/api/projects` — extended with board_layer, project_layer_links
- `/api/tasks` — routing triggers on status change
- `/api/retainers` — new CRUD endpoints
- `/api/client-users` — invite, deactivate, list

## Key Interactions to Verify
- Client user logs in via magic link → sees only their client's deliverables
- Internal user moves task to `client_review` → task appears in client portal
- Client approves deliverable → internal + master projects updated
- Client rejects deliverable → revision task created on internal board
- Client comments on deliverable → action item created internally with assignee alert
- Retainer rollup shows correct hours/budget percentage with color coding

## Edge Cases
- Client user tries to access internal project → 403
- Deactivated client_user tries to log in → rejected
- Task synced to client project when no client project linked → graceful no-op
- Retainer period ends → rollup resets for new period
- Task deleted from internal board → synced client task also removed
- Concurrent approval (two client users approve same deliverable simultaneously)
- Magic link used after expiry → clear error message

## Critical Paths (E2E)
1. Full approval flow: internal creates task → moves to client_review → client sees it → client approves → all layers updated
2. Full rejection flow: client rejects → revision task created → assignee notified → assignee fixes → re-submits to client
3. Portal access control: client_user can ONLY see client-layer projects, cannot navigate to any internal URL
